# Experiment Results — 2026-02-25

## Image
`ghcr.io/alicia-lyu/geodb-dbtoaster:latest`

## Command
```bash
docker pull ghcr.io/alicia-lyu/geodb-dbtoaster:latest
docker run --rm ghcr.io/alicia-lyu/geodb-dbtoaster:latest
```

## Results

### Unlimited (succeeded)
- Warmup: 400,000 customers
- Memory after warmup: **4,775,296 KB (~4.55 GB)**
- Maintenance measured for: 49,999 customers
- Maintenance time: **2.38459 s**
- Average time per update: **47.69 µs**

Notable memory growth steps during warmup:
| Customers | Memory (KB) | Notes |
|-----------|-------------|-------|
| 1,000 | 1,152,128 | Start |
| 66,000 | 2,035,456 | +883 MB jump |
| 132,000 | 2,939,136 | +882 MB jump |
| 263,000 | 4,418,944 | +1.4 GB jump |
| 400,000 | 4,775,296 | Warmup complete |

### Memory-constrained runs (all failed with `std::bad_alloc`)
| ulimit -v (KB) | Crashed at | Memory at crash |
|----------------|-----------|-----------------|
| 2,800,000 | ~131k customers | ~2,232,884 KB |
| 2,400,000 | ~131k customers | ~2,232,824 KB |
| 2,000,000 | ~65k customers | ~1,683,544 KB |
| 1,600,000 | ~48k customers | ~1,596,256 KB |

All constrained runs crash well before the memory limit is actually reached — the allocator aborts when it cannot satisfy a large batch allocation (the big memory jumps seen in the unlimited run).
